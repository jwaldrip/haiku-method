The Problem

AI assistants are powerful but chaotic

Without structure, every AI session is a blank page. Work gets lost between conversations. There are no quality gates, no lifecycle, no way to know where you left off.

Context Evaporation

The AI forgets everything between sessions. Progress is lost. Decisions are remade. Mistakes are repeated.

No Quality Gates

The AI can declare its work "done" without tests passing, without review, without evidence. Trust is unearned.

No Domain Awareness

Every AI session uses generic instructions. Security work gets software patterns. Documentation gets code review gates.

Origin

AI-DLC: the first attempt

Map the software development lifecycle to AI-assisted workflows. Give agents roles (hats), structure (phases), and persistence (state files).

The v1.0 Pipeline

Elaborate → Execute → Review → Operate

Hard-coded. Software-only. One pipeline fits all.

What Worked

  • Hat-based roles — planner, builder, reviewer
  • Completion criteria — verifiable exit conditions
  • Backpressure — hooks block the agent until gates pass
  • Persistent state — survives context resets

What Didn't

  • Software-only — no path for design, product, security
  • Fixed pipeline — can't reorder or skip phases
  • Global hats — same roles for every kind of work
  • Single review model — one size fits none

Software isn't the only discipline

Users needed design passes, product spec passes, security audit passes. The hard-coded pipeline assumed one discipline. Reality demanded cross-functional iteration.

Core Mechanic

The Stage Loop

Every stage — regardless of studio or domain — executes the same five-step cycle. The stage's metadata drives behavior; the loop is universal.

1. Decompose

Break stage into units

2. Execute

Bolt loop per unit

3. Review

Adversarial agents

4. Persist

Save scoped outputs

5. Gate

auto / ask / external

Decompose

Load inputs from prior stages. Break the work into units with completion criteria and a dependency DAG. If upstream outputs are insufficient, surface the gap to the human — stage-backs are always human-initiated.

Execute (Bolt Loop)

For each unit in DAG order, cycle through the stage's hat sequence. Each hat runs in isolation. Quality gates enforce standards at each cycle. If criteria aren't met, increment the bolt counter and retry.

Stage Loop — Step 3

Adversarial Review Agents

After execution, specialized review agents evaluate the stage's work in parallel. Each agent has a specific mandate and checklist — they catch what the builder's hat cannot see.

Cross-Stage Includes

Stages can include review agents from other stages via review-agents-include. The development stage pulls in design's consistency agent to verify the implementation respects the design intent.

Domain-Specific

Each studio's stages define review agents appropriate to their domain. Compliance has thoroughness and evidence-quality. Incident response has severity-accuracy and safety. No one-size-fits-all.

Stage Loop — Step 5

Review Gates

Every stage ends with a gate that controls advancement. The gate type is declared per-stage in the studio definition.

auto

Advances when quality gates pass. No human in the loop.

Examples: inception, triage, research

ask

Pauses for human approval. The primary control point.

Examples: design, development, draft

external

Pushes to PR/MR, blocks for external reviewer approval.

Examples: product, security, certify

await

Work is done but blocks for an external event (customer reply, CI result).

Examples: proposal, negotiation, mitigate

Modes

Continuous vs Discrete

Both modes run the same stage loop. The difference is what happens after a review gate passes.

BehaviorContinuousDiscrete
Stage sequenceSame — studio's stage listSame
Stage loopSame 5-step cycleSame
Gate: autoAdvance and run next stageStop, wait for user
Gate: askPause, then advance and continuePause, then stop
Gate: externalBlock, then advance and continueBlock, then stop
Gate: awaitBlock for event, then advance and continueBlock for event, then stop

Continuous mode never skips or collapses stages

Every stage runs its own elaborate, execute, adversarial review, and review gate cycle. Continuous mode simply controls whether the agent automatically proceeds to the next stage or hands control back to the user.

Quality Model

Backpressure, not prescription

H·AI·K·U doesn't tell the agent how to build. It defines constraints that must be satisfied. The agent iterates until all gates clear. Quality enforcement is structural, not behavioral.

Quality Gates (Machine)

Run automatically via Stop hook. The agent cannot override, weaken, or skip them.

  • Tests — npm test
  • Lint — npm run lint
  • Typecheck — tsc --noEmit
  • Build — npm run build

An agent that writes failing code cannot declare the unit done.

Adversarial Review (Judgment)

Specialized review agents verify criteria that machines can't check.

  • Correctness — does it match the spec?
  • Security — any injection vectors?
  • Accessibility — WCAG compliance?
  • Architecture — follows patterns?

The builder never reviews its own output.

Hat Isolation

Each hat executes in a fresh context. The planner's reasoning doesn't leak into the builder. The builder's assumptions don't influence the reviewer. This isolation is adversarial by design.

Delivery

One PR per intent

The intent is the atomic deliverable. Units and bolts are internal iteration within the intent branch — they don't produce separate PRs.

Git Persistence Model

main
  └─ haiku/my-feature/main ← intent branch (worktree)
      ├─ stage: inception (commits)
      ├─ stage: design (commits)
      ├─ stage: development (commits)
      └─ PR → main ← one PR for the whole intent

Why One PR

  • Units are internal elaboration, not review boundaries
  • Per-unit PRs fragment review and create noise
  • The intent branch captures the full narrative
  • Reviewers see the complete picture, not fragments

Persistence Adapters

  • Git — branches, commits, PRs (software, data-pipeline, migration...)
  • Filesystem — local snapshots, no VCS (ideation, sales, marketing...)

Same stage loop, different storage backend.

Philosophy

What we believe

Single-pass is the ideal

All disciplines collaborating simultaneously. One hive mind. No handoffs. Multi-stage exists as a concession for teams with siloed expertise — not because it's better.

Design is a thinking tool

Design artifacts are elaboration outputs, not construction deliverables. They emerge from collaborative exploration, not a separate "design phase."

Backpressure over prescription

Don't tell the agent how to build. Define constraints that must be satisfied. The agent iterates until all gates clear. Quality is structural, not behavioral.

Metadata drives behavior

No conditional logic per stage. The stage definition's metadata constrains what the agent does. Same orchestration code, different behavior. The studio is the configuration.
