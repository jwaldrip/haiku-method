Where We Started

AI-DLC was born from frustration

AI assistants were powerful but chaotic. Without structure, every session was a blank page. Work was lost between conversations. There was no lifecycle, no quality gates, no way to know where you left off.

The Original Vision

Map the software development lifecycle to AI-assisted workflows. Give agents roles (hats), structure (phases), and persistence (state files). Call it AI-DLC — the AI Development Lifecycle.

What We Built (v1.0)

  • Elaborate — gather intent, decompose into units
  • Execute — build with planner/builder/reviewer hats
  • Review — multi-agent code review
  • Operate — post-deployment lifecycle

Lesson 1

Software isn't the only discipline

Users wanted design passes, product passes, security passes. The hard-coded elaborate→execute pipeline assumed software development. Reality demanded cross-functional iteration.

Lesson 2

Passes became stages, and stages need hats

Passes were just typed iterations — but they needed their own roles, review gates, inputs, and outputs. A "design pass" needed a designer hat and a design-reviewer hat, not the builder hat. The pass model evolved into a full stage model with a collapsible FSM.

The Stage Architecture

ConceptWhat It IsWhat It Replaced
StudioNamed lifecycle templateHard-coded pipeline
StagePhase with its own hats + review gatePasses
HatBehavioral role scoped to a stageGlobal hat definitions
Review Gateauto / ask / external per stageSingle review step

Key insight: single-stage collapses to the old behavior. The FSM is additive, not breaking.

Lesson 3

The lifecycle isn't just for code

Once we had stages and hats as composable primitives, the methodology stopped being software-specific. Sales has a lifecycle. Marketing has a lifecycle. Incident response, compliance, customer success — they all follow the same pattern: focused phases, specialized roles, quality gates between transitions.

Philosophy

What we believe now

Single-pass is the ideal

All disciplines collaborating simultaneously during elaboration. One hive mind. No handoffs. Multi-stage exists as a concession for teams with siloed SMEs — not because it's better.

Design is a thinking tool

Design artifacts are elaboration outputs, not construction deliverables. They emerge from collaborative exploration, not a separate "design phase."

Backpressure over prescription

Don't tell the agent how to build. Define constraints that must be satisfied — tests pass, types check, criteria met. The agent iterates until all gates clear.

Metadata drives behavior

No conditional logic per stage. The stage definition's metadata (hats, review mode, unit types) constrains what the agent does. Same code, different behavior.
