Prologue: Meet the Cast

Before the story begins, let's meet the players. Every character has a role. Nobody works alone.

🧑

You (Human)

The decision-maker. The visionary.

  • You provide the vision — what to build and why
  • You make decisions at key moments
  • You unblock when the AI gets stuck
  • You approve the final result before it ships
🎯

The Orchestrator

The project manager. Always running.

  • The AI that manages the whole process
  • Asks you questions until it truly understands
  • Breaks work into pieces and coordinates specialists
  • Decides which unit to build next based on dependencies
🔨

The Builder

The craftsperson. Heads down, writing code.

  • A specialist AI that writes code in a safe sandbox
  • Implements one piece at a time, following the plan
  • Runs tests and fixes failures as it goes
  • Commits clean, working code to a feature branch
🔍

The Reviewer

The quality inspector. Tough but fair.

  • A specialist AI that checks the Builder's work
  • Verifies every success criterion with concrete evidence
  • Sends work back with specific feedback if criteria aren't met
  • Only signs off when everything passes — no rubber stamps
⚙️

The System

The invisible hand. Always watching.

  • Automated hooks that run silently in the background
  • Saves progress so nothing is lost between sessions
  • Enforces quality gates and workflow rules
  • Monitors memory usage and triggers context recovery
🚧

Quality Gates

The tollbooths. No shortcuts allowed.

  • Automated checkpoints between phases
  • Tests, type checks, and linting must ALL pass to proceed
  • If any gate fails, the AI must fix the issue first
  • Three gates: Plan, Quality, and Criteria

Why Specs Matter

The difference between hoping for the best and knowing what done looks like.

🎲

Vibe Coding

the bad way
  • Human: "Build me a login page"
  • AI builds... something
  • Human: "No, not like that"
  • AI rebuilds... differently
  • Human: "Closer, but the styling is wrong"
  • Repeat forever
Problem: No definition of "done." The AI guesses. You react. Progress is random.
🔀 Tangled loop — going in circles
📋

Traditional Specs

the old way
  • PM writes a 40-page requirements doc
  • Developer reads it (maybe)
  • Developer builds what they understood
  • QA finds gaps between spec and implementation
  • Spec is already stale by the time it's reviewed
Problem: Specs and code live in separate worlds. They drift apart immediately.
↕️ Two parallel lines that diverge
🎯

AI-DLC Specs

the AI-DLC way
  • Human and AI define intent together — conversation, not document
  • Success criteria are measurable and verifiable — tests, type checks, performance thresholds
  • The AI builds to satisfy the spec — criteria are the only way it knows when it's done
  • Quality gates enforce the spec automatically — if criteria aren't met, work continues
  • Specs live alongside code in .ai-dlc/ — versioned, committed, always current
Result: The spec IS the contract. Clear spec = autonomous AI. Vague spec = stuck AI.
✅ Straight arrow from spec to working code
The AI-DLC insight: Autonomy is a function of criteria clarity.
Autonomy = f(Criteria Clarity)
  • Vague criteria ("make it look good") → AI keeps asking you questions, can't work alone
  • Clear criteria ("all tests pass, p95 < 200ms, WCAG AA contrast") → AI iterates autonomously until done

This is why AI-DLC spends so much time on elaboration. The planning phase isn't overhead — it's the thing that makes everything else work.

See the difference

Good criteria are the ones an AI can check without asking you.

❌ Bad criterion
"The login page should be fast"
What does "fast" mean? The AI can't verify this.
✅ Good criterion
"Login API responds in <200ms at p95 under 100 concurrent users"
Measurable. The AI runs a load test and checks.
❌ Bad criterion
"The UI should look nice"
The AI has no idea what "nice" means to you.
✅ Good criterion
"All pages pass WCAG AA contrast (4.5:1 body, 3:1 large text), use design system tokens only, and render correctly at 375px, 768px, and 1280px"
Verifiable. The AI checks contrast ratios, validates token usage, and tests breakpoints.

Outer Loop: The Assembly Line

Units flow through a pipeline. Independent units can build in parallel. Each one unlocks the next.

Unit 1: OAuth Setup
✅ Done
Unit 2: Callback
🔨 Building
Unit 3: Sessions
⏳ Waiting (needs Unit 1)
Unit 4: Login UI
⏳ Waiting (needs 2 & 3)

The dependency graph (DAG) determines the order. Unit 1 has no dependencies, so it starts first. Once Unit 1 completes, Units 2 and 3 can build in parallel — they're independent of each other. Unit 4 waits for both 2 and 3 before it starts. The AI manages all of this automatically.

☕
You
Watching. Or away doing something else. Available if needed.
🔨
The Builder
Writing code, running tests, committing changes. Fully autonomous.

Middle Loop: The Hat Rotation

Inside each unit, the AI cycles through hats. Each hat has one job. Quality gates stand between them.

📋 Planner
Creates the implementation plan
🔨 Builder
Writes code, runs commands
🚧 Quality Gates
Tests pass? Types OK? Lint clean?
🔍 Reviewer
Verifies every criterion with evidence
✅ Pass
Unit complete!
→ Next unit
❌ Fail
Feedback given
→ Back to Builder

When the Reviewer passes:

The unit is marked complete. Its branch is merged. The orchestrator picks the next ready unit from the dependency graph and starts a fresh hat rotation.

When the Reviewer fails:

The Reviewer writes specific, actionable feedback — not vague complaints. "Test for expired tokens is missing" not "needs more tests." The Builder reads this feedback and iterates. The cycle continues until every criterion passes.

What does a "hat" actually mean?

Think of hats like roles in a play. The same AI plays every part, but it can only wear one hat at a time. When wearing the Planner hat, it thinks about structure and approach. When wearing the Builder hat, it writes code. When wearing the Reviewer hat, it critically examines work against success criteria. The workflow-guard hook enforces this — if the AI tries to write code while wearing the Planner hat, the system stops it.

If the Reviewer finds problems, it writes specific feedback and the AI switches back to the Builder hat. This cycle repeats until the Reviewer is satisfied that every success criterion is met with evidence. No shortcuts. No "looks good to me." Concrete proof for every claim.

Inner Loop: The Bolt (Context Recovery)

This is the clever part. Here's the problem AI-DLC solves:

AI agents have limited memory — called a "context window." When the memory fills up, a normal AI would forget everything. AI-DLC solves this by saving progress to files on disk. When a new session starts, the AI reads those files and picks up exactly where it left off. This is called a bolt — one focused work session.

🟢 AI starts working
100%
🟢 Writing code, running tests
75%
🟡 Context getting low
50%
🟠 Warning! Save your work
35% ⚠️
🔴 Context critical!
25% 🚨
💾 AI saves state to files — session ends
8%
🔄 New session starts — fresh context!
100%
📂 AI loads state from files — continues exactly where it left off
100%
⚙️ System
enforce-iteration.sh fires when a session ends. It checks what work remains. If units are still in progress, it tells the next session to call /execute to continue. The AI never "forgets" mid-task.

How do these three loops work together?

Picture it like this: the assembly line picks the next unit. The hat rotation builds that unit through plan, build, review cycles. And within each hat's work session, the bolt ensures that even if the AI's memory runs out, progress is never lost.

A single feature might involve 4 units, each going through 2–3 hat rotations, each rotation spanning 1–3 bolts. That's a lot of autonomous work — potentially hours of building — all from one planning conversation.

Two systems keep everything on track: quality gates that enforce standards, and hooks that run silently in the background.

Three hard gates stand between hats. The AI cannot bypass them:

1. Plan Gate

Planner must save a complete implementation plan before the Builder can start.

"No building without a blueprint."

2. Quality Gate

Tests + types + lint must all pass before the Reviewer looks at the code.

"No review of broken code."

3. Criteria Gate

Every success criterion verified with concrete evidence before marking the unit done.

"No shipping without proof."

The Safety Net — Hooks

Hooks are automated scripts that fire at specific moments during a session. They run silently. You never see them. But they keep everything honest.

Why all this machinery?

AI is powerful but imperfect. Without guardrails, an AI might skip tests, write code outside its current task, or lose track of progress when a session ends. These hooks and gates create a safety net — not to slow things down, but to keep the AI honest and productive. Think of them as the rules of the road that make autonomous driving possible.

Act 5: Finishing and Learning

All units are done. The code is written. The tests pass. But the story isn't over yet. Three stages remain before the cycle is complete.

This is where individual units become a complete feature, where the feature becomes a deliverable, and where the whole team (human and AI) gets smarter for next time.

1

Integration Check

All the individual pieces are done. Now they need to work together.

The merged code is validated as a whole. Cross-unit tests run. Deployment checks fire. Monitoring consistency is verified. Does the callback handler actually connect to the session manager? Does the login UI correctly call the callback endpoint?

If issues are found, specific units get sent back for rework — not the entire feature. The system knows which unit owns which code, so it targets feedback precisely.

2

Delivery

🧑 You
Review the PR. Check the diffs. Read the summary. The PR description includes everything — what changed, why, how it was tested, and links to each unit's criteria verification. Then approve and merge.
3

Reflection

This is what separates AI-DLC from "just running an AI." The system learns from every cycle.

🎯 Orchestrator
Here's what I learned from this cycle:
Metrics: 4 units, 12 iterations, 3 reviewer round-trips
Friction point: The builder had trouble with OAuth callback — the test fixtures for mock OAuth servers were missing from the project
What worked: The TDD-style approach for session management produced clean, well-tested code on the first reviewer pass
Recommendations:
  • Add eslint-plugin-security to quality gates
  • Create a shared OAuth mock server fixture for future auth work
  • Consider the adversarial workflow for security-sensitive features
🧑 You
You validate the AI's analysis. Did it identify the real friction points? Are the recommendations useful? Then choose what's next:
Apply recommendations Start a follow-up Archive and close
Behind the scenes
The reflection produces reflection.md and settings-recommendations.md in .ai-dlc/{slug}/. "Apply recommendations" auto-applies the suggested config changes to your project. "Start a follow-up" uses /followup to create a new intent that iterates on what was learned — carrying forward all context from this cycle. "Archive and close" captures organizational memory and cleans up the work state.

Why reflection matters: Each cycle makes the next one better. The AI remembers what worked, what didn't, and adapts. Over time, your project's AI-DLC configuration becomes tuned to your specific codebase, team patterns, and quality standards. The first feature takes the longest. Every subsequent one is faster and smoother.

Epilogue: Other Ways In

The full cycle isn't the only path. Sometimes you need a shortcut. Sometimes you need to pick up where you left off.

/quick

For tiny fixes — typos, config changes, one-liners. Skip everything. Just fix it and commit. No state files, no worktrees, no ceremony.

Example: /quick fix the typo in README.md

/autopilot

For well-understood features. The AI runs the entire cycle (elaborate, build, deliver) with guardrails: pauses on ambiguity, pauses if more than 5 units, and pauses before creating a PR.

Example: /autopilot Add dark mode toggle to settings

/resume

Pick up where you left off. Finds your previous work from files or git branches and restores the session. Perfect for the morning after.

Example: /resume oauth-login

/operate

For post-deployment work. Manage scheduled jobs, runbooks, and operational tasks. The feature is live — now keep it running.

Example: /operate my-feature --status

Every entry point leads to the same core system. The difference is how much ceremony you want. A typo fix doesn't need a plan. A new feature does. AI-DLC scales the process to match the work.
