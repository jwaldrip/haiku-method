H·AI·K·U Architecture

A domain-agnostic work orchestrator — studios, stages, and accumulated knowledge

H·AI·K·U is a lifecycle orchestration system. It models any kind of work — software development, marketing campaigns, hardware engineering, legal review — as a pipeline of stages. Each stage plans its work, builds it, reviews it, and produces artifacts that feed the next stage.

The core insight: all structured work flows through stages of planning, building, and reviewing. A marketing team creating a campaign goes through research, creative, copy, review, and publish. A software team goes through inception, design, product, development, operations, and security. A hardware team goes through requirements, PCB design, firmware, and integration testing. The pattern is the same. The domain is the variable.

The default studio — ideation — provides a universal lifecycle that works for any discipline: research → create → review → deliver. Domain-specific studios (software, marketing, hardware) specialize this with their own stages, hats, and persistence layers.

Four concepts make this work:

Studio
A named lifecycle for a specific domain. The studio defines the stage order and the persistence layer — how work is saved, versioned, and delivered. Git for software. Notion for content. Filesystem for anything.
Stage
A lifecycle phase that plans, builds, and reviews its own work. Each stage defines its roles (hats), review mode, and outputs. One definition file contains everything.
Persistence
The adapter that saves, versions, and delivers work. Git (branches, commits, PRs) is the software adapter. Other adapters serve other domains. The studio declares its type — the stages don't care.
Knowledge
Two layers of accumulated context. The global pool persists project-level understanding across intents. Intent artifacts accumulate as stages complete. Every stage reads both.

The user chooses per-intent whether to run continuous (autopilot drives stages, user reviews at gates) or discrete (user invokes each stage). Both follow the same pipeline. The stages, artifacts, and quality are identical either way.

Full Hierarchy

Studio — named lifecycle for any domain (software, marketing, hardware, ...)
├── Persistence — how work is saved (git, notion, filesystem, ...)
└── Stage — lifecycle phase: plan → build → review
├── STAGE.md — hats, guidance, review mode, requires/produces
├── Review Gate — auto | ask | external
└── Unit — discrete piece of work
└── Bolt — one cycle through the stage's hat sequence
