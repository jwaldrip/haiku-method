How It Works

What is a “Hat”?

A hat is a set of injected instructions that tells a fresh AI agent how to behave. Here’s what actually happens:

1 Claude spawns a fresh agent
Clean context, no prior baggage
2 The system injects hat instructions
A markdown file that defines the role’s behavior, rules, and quality gates
3 The agent works according to its hat
It only knows how to be a Builder, or a Reviewer, etc.

Think of it like hiring a contractor: you bring someone in (fresh agent), hand them a job description (hat instructions), and they do exactly that job. When the job’s done, they leave. The next job gets a new person with a new job description.

What’s in a hat file?

Each hat is a markdown file (plugin/hats/{hat-name}.md) that defines:

  • What the agent MUST do (required steps)
  • What the agent MUST NOT do (boundaries)
  • Quality gates it must pass before finishing
  • When to call /advance or /fail
Why fresh agents?

Each hatted agent starts with a clean context window. This means:

  • No confusion from previous units or hats
  • Full context budget for the current task
  • Failures in one unit don’t bleed into another

Why Specs Matter

The difference between hoping for the best and knowing what done looks like.

🎲

Vibe Coding

the bad way
  • Human: "Build me a login page"
  • AI builds... something
  • Human: "No, not like that"
  • AI rebuilds... differently
  • Human: "Closer, but the styling is wrong"
  • Repeat forever
Problem: No definition of "done." The AI guesses. You react. Progress is random.
🔀 Tangled loop — going in circles
📋

Traditional Specs

the old way
  • PM writes a 40-page requirements doc
  • Developer reads it (maybe)
  • Developer builds what they understood
  • QA finds gaps between spec and implementation
  • Spec is already stale by the time it's reviewed
Problem: Specs and code live in separate worlds. They drift apart immediately.
↕️ Two parallel lines that diverge
🎯

AI-DLC Specs

the AI-DLC way
  • Human and AI define intent together — conversation, not document
  • Success criteria are measurable and verifiable — tests, type checks, performance thresholds
  • The AI builds to satisfy the spec — criteria are the only way it knows when it's done
  • Quality gates enforce the spec automatically — if criteria aren't met, work continues
  • Specs live alongside code in .ai-dlc/ — versioned, committed, always current
Result: The spec IS the contract. Clear spec = autonomous AI. Vague spec = stuck AI.
✅ Straight arrow from spec to working code
The AI-DLC insight: Autonomy is a function of criteria clarity.
Autonomy = f(Criteria Clarity)
  • Vague criteria ("make it look good") → AI keeps asking you questions, can't work alone
  • Clear criteria ("all tests pass, p95 < 200ms, WCAG AA contrast") → AI iterates autonomously until done

This is why AI-DLC spends so much time on elaboration. The planning phase isn't overhead — it's the thing that makes everything else work.

See the difference

Good criteria are the ones an AI can check without asking you.

❌ Bad criterion
"The login page should be fast"
What does "fast" mean? The AI can't verify this.
✅ Good criterion
"Login API responds in <200ms at p95 under 100 concurrent users"
Measurable. The AI runs a load test and checks.
❌ Bad criterion
"The UI should look nice"
The AI has no idea what "nice" means to you.
✅ Good criterion
"All pages pass WCAG AA contrast (4.5:1 body, 3:1 large text), use design system tokens only, and render correctly at 375px, 768px, and 1280px"
Verifiable. The AI checks contrast ratios, validates token usage, and tests breakpoints.

Deep Dive: Domain Discovery — What the Explorer Actually Finds

What the Discovery Subagent Explores

The discovery adapts to project maturity: greenfield projects get lighter exploration focused on establishing conventions. Established codebases get thorough pattern analysis to ensure consistency.

Deep Dive: Unit Spec Format — What's in Each Unit File

Each unit is defined as a markdown file with YAML frontmatter. Here's the anatomy of a unit spec.

Frontmatter Schema

FieldTypeDescription
titlestringHuman-readable unit name
slugstringMachine-friendly identifier (e.g., unit-01-oauth-setup)
disciplineenumOne of: frontend, backend, infrastructure, observability, documentation, design
depends_onstring[]List of unit slugs this unit depends on
workflowstringOverride the intent-level workflow for this unit
statusenumpending | ready | in_progress | completed | blocked

Body Sections

  • Description — What this unit accomplishes and why
  • Completion Criteria — Unit-specific measurable exit conditions
  • Technical Notes — Implementation guidance, patterns to follow
  • Operations — Optional deployment, monitoring, and operational blocks

Dependency Rules

  • Dependencies form a DAG — no circular references allowed
  • A unit becomes ready only when all its depends_on units are completed
  • Units with no dependencies are ready immediately
  • Independent units at the same DAG depth can execute in parallel

Deep Dive: Branch Strategies — How Code Gets Delivered

AI-DLC supports three branching strategies, each suited to different team workflows.

Unit Strategy (Separate PRs)

main unit-01-branch PR → merge

Each unit gets its own branch and PR. Units merge independently as they complete. Best for code review granularity and incremental delivery.

Intent Strategy (One PR)

main intent-branch unit branches merge here single PR

All unit branches merge into a shared intent branch. One big PR at the end. Best when you want to review the complete feature as a whole. Note: the intent strategy implies auto-merge of unit branches into the intent branch.

Trunk Strategy (Direct Commits)

main direct commits

No branches, no PRs. Code commits directly to the main branch. Best for solo developers or high-trust environments with strong automated testing.

Deep Dive: DAG Execution — How Dependencies Drive the Pipeline

The dependency graph isn't just a visual aid — it's the engine that drives execution order.

Status Transitions

pending ready in_progress completed

or: in_progress → blocked

Execution Rules

  • Units declare depends_on in their frontmatter, referencing other unit slugs
  • A unit becomes ready when all its dependencies are completed
  • /execute picks the next ready unit from the DAG automatically
  • If a unit is blocked, work continues on other ready units in parallel
  • The enforce-iteration.sh hook checks DAG status at session end and suggests next steps
  • The lib/dag.sh library provides the DAG traversal and status management functions

Wave-Based Parallel Execution

When using agent teams (parallel mode), units at the same DAG depth form a "wave." All units in a wave can execute simultaneously:

  • Wave 1: All units with no dependencies (root nodes)
  • Wave 2: Units whose dependencies were all in Wave 1
  • Wave 3: And so on, until all units are scheduled

Each wave's units are assigned to separate agent teammates. When a wave completes, merge handling reconciles the branches before the next wave begins.

Middle Loop: The Hat Rotation

Inside each unit, the AI cycles through hats. Each hat has one job. Quality gates stand between them.

📋 Planner
Creates the implementation plan
🔨 Builder
Writes code, runs commands
🚧 Quality Gates
Tests pass? Types OK? Lint clean?
🔍 Reviewer
Verifies every criterion with evidence
✅ Pass
Unit complete!
→ Next unit
❌ Fail
Feedback given
→ Back to Builder

When the Reviewer passes:

The unit is marked complete. Its branch is merged. Claude picks the next ready unit from the dependency graph and starts a fresh hat rotation.

When the Reviewer fails:

The Reviewer writes specific, actionable feedback — not vague complaints. "Test for expired tokens is missing" not "needs more tests." The Builder reads this feedback and iterates. The cycle continues until every criterion passes.

What does a "hat" actually mean?

Think of hats like roles in a play. The same AI plays every part, but it can only wear one hat at a time. When wearing the Planner hat, it thinks about structure and approach. When wearing the Builder hat, it writes code. When wearing the Reviewer hat, it critically examines work against success criteria. The workflow-guard hook enforces this — if the AI tries to write code while wearing the Planner hat, the system stops it.

If the Reviewer finds problems, it writes specific feedback and the AI switches back to the Builder hat. This cycle repeats until the Reviewer is satisfied that every success criterion is met with evidence. No shortcuts. No "looks good to me." Concrete proof for every claim.

Deep Dive: Hat Details — What Each Role Actually Does

Each hat follows a precise protocol. Here's exactly what happens in each one.

📋 Planner

  1. Reads the unit's completion criteria and any previous iteration notes
  2. Checks connected providers for relevant context (designs, specs, tickets)
  3. Creates a tactical implementation plan with specific file changes
  4. Saves the plan to current-plan.md in the state directory
  5. Cannot proceed until the plan is written — the Plan Gate enforces this

🔧 Builder

  1. Loads the plan from current-plan.md
  2. Implements changes incrementally, committing after each logical step
  3. Runs backpressure checks after each change (tests, types, lint)
  4. If a check fails, the Node Repair Operator kicks in with an escalation chain:
    • RETRY — Try the same approach again with fixes
    • DECOMPOSE — Break the change into smaller steps
    • PRUNE — Revert the failing change and try an alternative
    • ESCALATE — Ask the human for help (blocker)
  5. Produces deployment, monitoring, and operational artifacts alongside code

🔍 Reviewer

The review follows three stages:

  1. Stage 1: Spec Compliance (CoVe Pattern) — For each completion criterion, the reviewer formulates verification questions, independently checks the evidence, and compares findings against the criterion. This "Chain of Verification" prevents self-confirmation bias.
  2. Stage 2: Code Quality — Checks for code smells, security issues, performance concerns, test coverage, and adherence to project patterns discovered in the discovery phase.
  3. Stage 3: Operational Readiness (Conditional) — Only runs if the unit has operational requirements. Checks deployment manifests, monitoring configuration, and runbook completeness.

The reviewer can delegate parallel sub-reviews. Findings are scored:

  • High — Blocks advancement. Must be fixed.
  • Medium — Warns but does not block. Should be addressed.
  • Low — Informational. Noted but ignored for advancement.

Inner Loop: The Bolt (Context Recovery)

This is the clever part. Here's the problem AI-DLC solves:

AI agents have limited memory — called a "context window." When the memory fills up, a normal AI would forget everything. AI-DLC solves this by saving progress to files on disk. When a new session starts, the AI reads those files and picks up exactly where it left off. This is called a bolt — one focused work session.

Deep Dive: State Management — How Nothing Gets Lost

AI-DLC uses a two-tier persistence model that separates permanent project artifacts from temporary working state.

Committed Artifacts

Checked into version control. Survives everything.

.ai-dlc/{intent-slug}/

  • intent.md
  • unit-01-*.md, unit-02-*.md, ...
  • completion-criteria.md
  • discovery.md
  • settings.yml
Ephemeral State

Gitignored. Survives /clear but not branch switches.

.ai-dlc/{intent-slug}/state/

  • iteration.json
  • scratchpad.md
  • blockers.md
  • current-plan.md

Why two tiers? Committed artifacts define what needs to be built and how to verify it. Ephemeral state tracks where you are in the process. The artifacts are the source of truth; the state is the bookmark. If ephemeral state is lost, /resume can reconstruct it from the committed artifacts and git history.

Deep Dive: Agent Teams — Parallel Execution Mode

When multiple units are ready simultaneously, AI-DLC can spawn parallel agent teams to execute them concurrently.

How It Works

  1. Wave detection — The DAG is analyzed to identify groups of units at the same depth with all dependencies satisfied
  2. Team creation — Each unit in the wave gets assigned to a separate agent teammate
  3. Teammate spawning — Sub-agents are spawned with full AI-DLC context injected via subagent-hook.sh
  4. Parallel execution — Each teammate works independently on its unit, following the full hat rotation
  5. Merge handling — When all teammates complete, branches are merged and cross-unit integration is validated

Constraints

  • Only units with no dependency conflicts can execute in parallel
  • Each teammate operates in its own git worktree to avoid conflicts
  • The session agent monitors all teammates and handles failures
  • If a teammate fails, only its unit is affected — other teammates continue

Pass Transitions — How the System Manages Multi-Pass Execution

When all units in the current pass complete, the execute skill checks whether there's a next pass to transition to.

Transition Logic

All units complete Read passes array Find next pass Update active_pass

or: no next pass → intent complete → proceed to delivery

Step by Step

  1. Read the passes array from intent.md (e.g., [design, dev])
  2. Read active_pass (e.g., "design")
  3. Find the next pass in the array (e.g., "dev")
  4. If next pass exists:
    • Update active_pass to the next value
    • Do not mark intent complete
    • Tell the user to run /elaborate to define units for the new pass
  5. If no next pass (last pass done):
    • Mark intent complete
    • Proceed to delivery

Unit-Pass Relationship

Each unit's frontmatter includes a pass field that ties it to a specific pass. During execution, only units matching the active_pass are considered for the DAG. When a pass transition occurs, previously completed units from earlier passes remain accessible as context for elaborating the next pass's units.

How do these three loops work together?

Picture it like this: the assembly line picks the next unit. The hat rotation builds that unit through plan, build, review cycles. And within each hat's work session, the bolt ensures that even if the AI's memory runs out, progress is never lost.

A single feature might involve 4 units, each going through 2–3 hat rotations, each rotation spanning 1–3 bolts. That's a lot of autonomous work — potentially hours of building — all from one planning conversation.

Deep Dive: Hard Gates — Verification Logic

Each gate has exact verification logic that cannot be overridden by the AI.

PLAN_APPROVED

  • Checked when advancing from Planner to Builder
  • Verifies that current-plan.md exists in the state directory
  • Plan must contain actionable implementation steps, not just descriptions
  • If the plan is missing or empty, the advance is blocked

TESTS_PASS

  • Checked when advancing from Builder to Reviewer
  • Runs the project's test suite, type checker, and linter
  • All three must exit with code 0 (zero failures)
  • Commands are configurable via settings.yml quality gate overrides

CRITERIA_MET

  • Checked when advancing from Reviewer to complete
  • Every completion criterion must have associated evidence
  • Evidence can be: test output, metric measurement, screenshot, or manual verification
  • A single High-severity finding blocks advancement

Eight hooks form the automated safety system. Each fires at a specific point in the Claude Code lifecycle.

Deep Dive: /advance & /fail — The State Machine Transitions

These two skills are the gears of the hat rotation.

/advance — Move Forward

  1. Checks that the current hat's gate conditions are satisfied
  2. If gates pass, transitions to the next hat in the workflow sequence
  3. If the current hat is the last in the workflow, marks the unit as complete
  4. On completion: merges the unit branch using the configured strategy
    • Unit strategy: Opens a PR to main (or auto-merges if configured)
    • Intent strategy: Merges into the intent branch (auto-merge)
    • Trunk strategy: Commits are already on main
  5. Triggers integration validation if all units in the intent are now complete
  6. Updates DAG status so the session agent can pick the next unit

/fail — Go Back

  1. Records the failure reason and specific feedback in the state
  2. Returns to the previous hat (typically Reviewer → Builder)
  3. Increments the bolt iteration counter
  4. The Builder sees the Reviewer's feedback on the next session start via inject-context.sh
  5. If the same failure occurs repeatedly, the escalation chain activates (retry → decompose → prune → escalate to human)

Why all this machinery?

AI is powerful but imperfect. Without guardrails, an AI might skip tests, write code outside its current task, or lose track of progress when a session ends. These hooks and gates create a safety net — not to slow things down, but to keep the AI honest and productive. Think of them as the rules of the road that make autonomous driving possible.

Deep Dive: Integration Validation — The 10 Steps of /integrate

When all units are complete, /integrate runs a comprehensive validation of the combined output.

  1. Merged state verification — Confirms all unit branches are properly merged and no conflicts remain
  2. Backpressure re-run — Full test suite, type checking, and linting on the combined codebase
  3. Criteria cross-check — Verifies all intent-level completion criteria are satisfied (not just unit-level)
  4. Cross-unit integration tests — Runs tests that span multiple units (e.g., "does the UI actually call the API?")
  5. Emergent issue detection — Looks for issues that only appear when units are combined (naming conflicts, resource contention)
  6. Deployment manifest validation — Checks that deployment artifacts are consistent and complete
  7. Monitoring configuration check — Verifies that observability tooling covers all new code paths
  8. Operations readiness — Confirms runbooks and operational procedures are in place
  9. Dry-run simulation — Simulates the deployment process without actually deploying
  10. Decision gate — Presents the integration report to the human for final approval or targeted rework
Deep Dive: Operations — The /operate Lifecycle

After construction and delivery, /operate handles the post-deployment phase.

Operation Types

  • Scheduled operations — Recurring maintenance tasks (e.g., certificate rotation, dependency updates)
  • Reactive operations — Incident response, monitoring alerts, and bug triage
  • Process operations — Deployment procedures, rollback checklists, and release management

The /operate Flow

  1. List — Shows all defined operations and their status
  2. Execute — Runs a specific operation (agent-owned tasks run autonomously; human-owned tasks present checklists)
  3. Deploy manifests — Generates and validates deployment configuration
  4. Status monitoring — Checks deployment health, service status, and alert state
  5. Teardown — Clean decommissioning of resources when a feature is retired
Deep Dive: Reflection — How the AI Learns

The /reflect skill performs a structured analysis of the completed intent cycle.

What It Analyzes

  • Session transcripts — Conversation patterns, question frequency, decision points
  • Tool failures — Which commands failed, how often, and what was the recovery path
  • Retry loops — Where the Builder/Reviewer cycle repeated and why
  • Compound learnings — Patterns that emerge across multiple units (via /compound)

What It Produces

  • reflection.md — Narrative analysis of the cycle: metrics, friction points, what worked, what didn't
  • settings-recommendations.md — Concrete configuration changes suggested for future work (e.g., "add eslint-plugin-security to quality gates")

The Three Paths After Reflection

  • Apply recommendations — Auto-applies suggested config changes to your project settings
  • Start a follow-up (/followup) — Creates a new intent that iterates on what was learned, carrying forward all context from this cycle
  • Archive and close — Captures organizational memory, cleans up ephemeral state, and marks the intent as fully closed
