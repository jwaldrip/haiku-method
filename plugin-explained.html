AI-DLC

AI-Driven Development Lifecycle — a Claude Code plugin that structures AI-assisted software development into iterative, quality-gated workflows.

By The Bushido Collective v1.80.0 Apache-2.0 License ai-dlc.dev

! The Problem AI-DLC Solves

AI-assisted coding today is powerful but chaotic. Without structure, AI agents produce inconsistent, unverified work that drifts from intent.

  • Context window resets lose progress. Every /clear wipes the slate. Previous decisions, plans, and learnings vanish, forcing the AI to start from scratch.
  • No quality gates means errors propagate. Without enforcement, AI-generated code passes through unchecked. Bugs compound across iterations until the codebase is broken beyond easy repair.
  • Ambiguous success criteria lead to "good enough" work. When there is no definition of done, the AI guesses when to stop. The result: incomplete features, missing edge cases, skipped tests.
  • Traditional SDLC phases create friction, not quality. Agile ceremonies and heavyweight processes were designed for human teams. Applying them to AI agents adds overhead without improving output.
  • Organizational knowledge is scattered across systems. Design specs in Figma, tickets in Jira, context in Slack, docs in Confluence — the AI cannot see the full picture.

How AI-DLC Solves These

  • File-based state survives resets. All intent, unit, criteria, and iteration state is persisted to .ai-dlc/ on disk. Context resets are a feature, not a bug.
  • Backpressure gates enforce quality automatically. Tests, types, lint, and security checks must pass before advancing. The AI cannot skip quality.
  • Completion criteria define "done" precisely. Every unit has measurable, verifiable exit conditions written before work begins. No ambiguity.
  • Hat-based workflows match phase to behavior. Instead of human process ceremonies, the AI wears role-specific "hats" that shape its behavior for planning, building, reviewing, and more.
  • Provider integrations bridge external systems. Design, spec, ticketing, and communication tools are integrated via structured provider specs.

◆ Core Concepts

AI-DLC maps familiar Agile concepts onto an AI-native structure optimized for autonomous iteration.

The Hierarchy

Intent
Feature / Epic
Units
Tickets / Stories
Bolts
Sprints / Iterations
Hats
Roles / Behaviors

An Intent is the overall thing being built (like an Epic). It decomposes into Units (discrete work items, like tickets). Each unit is executed through Bolts (iteration cycles, like sprints). During each bolt, the AI wears different Hats that control its behavior (planner, builder, reviewer, etc.).

Three Operating Modes

👥
HITL
Human-in-the-Loop
Human validates each step. Maximum control. Best for high-stakes or unfamiliar work.
👁
OHOTL
Observed Human-on-the-Loop
Human watches in real-time, can intervene at any point. Balanced oversight with speed.
🤖
AHOTL
Autonomous Human-on-the-Loop
AI iterates autonomously, quality gates enforce standards. Human reviews results. Maximum velocity.

Backpressure Over Prescription

Key Principle: Don't tell the AI how to work. Define what must pass — tests, types, lint, security — and let the AI figure out how to satisfy the constraints. Backpressure gates create natural quality pressure without micromanaging implementation.

Completion Criteria Enable Autonomy

Autonomy = f(Criteria Clarity)

The clearer and more measurable the completion criteria, the more autonomously the AI can operate. Vague criteria require human judgment at every step. Precise, verifiable criteria allow the AI to self-validate and iterate independently.

▶ Four Phases

AI-DLC organizes work into four sequential phases, each with its own entry point and purpose.

🎯 Inception
🔨 Construction
🚀 Operations
📊 Reflection
🎯 Phase 1: Inception — Mob Elaboration

Entry point: /elaborate

The human provides an intent (what they want to build). The AI conducts a collaborative elaboration session:

  • Asks clarifying questions to sharpen the intent
  • Decomposes the intent into units with completion criteria
  • Performs domain discovery (/elaborate-discover) to understand the codebase
  • Syncs tickets to external systems (/elaborate-ticket-sync)
  • Generates wireframes for frontend units (/elaborate-wireframes)
  • Establishes a dependency DAG between units

Output: .ai-dlc/{intent-slug}/ directory with intent.md, unit-*.md, completion-criteria.md, and discovery.md

🔨 Phase 2: Construction — Autonomous Execution

Entry point: /execute

The core build phase where code gets written, tested, and reviewed:

  • DAG-based unit selection — picks the next ready unit respecting dependencies
  • Hat-based workflow cycles — each unit goes through a sequence of hats (e.g., planner → builder → reviewer)
  • Backpressure gates — tests, types, lint must pass to advance
  • Context resets embraced — state persists to filesystem; /clear is safe
  • Automatic iteration — if the reviewer finds issues, the workflow fails back to builder

The /advance skill moves to the next hat; /fail returns to the previous one. The enforce-iteration hook at session end checks DAG status and prompts continuation.

🚀 Phase 3: Operations — Post-Deployment

Entry point: /operate

Handles operational concerns after construction is complete:

  • Scheduled operations — recurring maintenance tasks
  • Reactive operations — incident response and monitoring
  • Process operations — deployment procedures and checklists
  • Agent-owned tasks — automated operations the AI executes
  • Human-owned tasks — checklists for manual steps (e.g., DNS updates, vendor config)

Generates deployment manifests and operational runbooks as structured artifacts.

📊 Phase 4: Reflection — Learning

Entry point: /reflect

After an intent is completed, the reflection phase captures organizational learning:

  • Analyzes execution metrics (bolts per unit, failure rates, time patterns)
  • Captures what worked and what didn't
  • Generates recommendations for future work
  • Feeds learnings back into organizational memory
  • Produces a structured reflection artifact

The /compound skill aggregates learnings across multiple intents for broader patterns.

How Context Resets Work

When the context window fills up, the human (or AI) runs /clear. All in-memory context vanishes. But all AI-DLC state — the current hat, iteration count, scratchpad notes, blockers, and plan — lives in .ai-dlc/{slug}/state/ on disk. When a new session starts, the inject-context.sh hook reads this state and reconstructs the full working context. The AI resumes exactly where it left off.

The Linear Flow

1 Session starts — inject-context.sh fires, reads .ai-dlc/ state, injects hat instructions + criteria + scratchpad
2 Hat shapes behavior — AI follows hat-specific instructions (plan, build, review, etc.)
3 Work executes — Code written, tests created, files modified
4 Quality check — Backpressure gates run (tests, types, lint, security)
5 Advance or fail — /advance to next hat, or /fail back to fix issues
6 Context fills — /clear wipes context window; state safe on disk
7 Resume — Return to step 1; state fully restored from filesystem
✓ Complete — All units done, /integrate validates cross-cutting concerns, deliver

⚓ Hook System

Hooks are shell scripts that fire at specific points in the Claude Code lifecycle. They provide the "enforcement without prescription" that powers AI-DLC.

Session Start
inject-context.sh
Loads state from filesystem. Injects the current hat instructions, intent description, completion criteria, scratchpad notes, blockers, and iteration number into the new session context. This is what makes context resets seamless.
Pre-Tool-Use (During Work)
redirect-plan-mode.sh
Intercepts EnterPlanMode tool calls and redirects them to /elaborate. Ensures planning happens through the AI-DLC elaboration flow rather than Claude's native plan mode.
Pre-Tool-Use
subagent-hook.sh
When the AI spawns sub-agents (via Task tool), this hook injects AI-DLC context into the sub-agent prompt so child agents are aware of the current intent, unit, and hat.
Pre-Tool-Use
prompt-guard.sh
Security hook. Scans .ai-dlc/ files for potential prompt injection patterns before they are loaded into context. Prevents malicious content in state files from hijacking the AI.
Pre-Tool-Use
workflow-guard.sh
Warns when the AI attempts file edits outside the scope of the current hat's context. Prevents builders from reviewing, reviewers from building, etc.
Post-Tool-Use (After Work)
context-monitor.sh
Monitors context window usage. Warns at 35% remaining and urgently at 25% remaining, prompting the user to /clear before context is exhausted and work is lost.
Session End (Stop)
enforce-iteration.sh
Fires when the AI stops. Checks DAG status for remaining work, auto-reconciles completed intents, and prompts the human about whether to continue with the next unit or stop. The "keep going" mechanism.
Sub-agent context: The subagent-context.sh script provides the context payload that subagent-hook.sh injects. This ensures multi-agent workflows maintain AI-DLC awareness across agent boundaries.

💾 State Management

AI-DLC uses a two-tier persistence model that separates permanent project artifacts from temporary working state.

Committed Artifacts

Checked into version control. Survives everything.

.ai-dlc/{intent-slug}/

  • intent.md
  • unit-01-*.md, unit-02-*.md, ...
  • completion-criteria.md
  • discovery.md
  • settings.yml
Ephemeral State

Gitignored. Survives /clear but not branch switches.

.ai-dlc/{intent-slug}/state/

  • iteration.json
  • scratchpad.md
  • blockers.md
  • current-plan.md
Why two tiers? Committed artifacts define what needs to be built and how to verify it. Ephemeral state tracks where you are in the process. The artifacts are the source of truth; the state is the bookmark. If ephemeral state is lost, /resume can reconstruct it from the committed artifacts and git history.

How State Survives /clear

Context fills up
User runs /clear
Memory wiped
inject-context.sh reads disk
Full context restored

◆ DAG-Based Execution

Units within an intent form a Directed Acyclic Graph (DAG) based on their dependencies. The execution engine respects this ordering.

Example Dependency Graph

unit-01-data-layer
completed
unit-03-api-routes
ready
unit-02-domain-model
in_progress (depends on unit-01)
unit-04-frontend
pending (depends on unit-02, unit-03)

Status Flow

pending
ready
in_progress
completed
or
in_progress
blocked

How It Works

  • Units declare depends_on in their frontmatter, referencing other unit slugs
  • A unit becomes ready when all its dependencies are completed
  • /execute picks the next ready unit from the DAG automatically
  • If a unit is blocked, work continues on other ready units in parallel
  • The enforce-iteration.sh hook checks DAG status at session end and suggests next steps
  • The lib/dag.sh library provides the DAG traversal and status management functions

🔌 Provider Integration

AI-DLC integrates with external tools through four provider categories. Each provider has a structured spec and JSON schema.

🎨 Design Providers
Figma, Adobe XD
Visual fidelity backpressure. Design specs, component tokens, and mockup references are pulled into the AI's context. The designer hat uses these to produce artifacts that match the design system.
📄 Spec Providers
Notion, Confluence, Google Docs
Requirements sync. Product specs, PRDs, and technical docs are referenced during elaboration to ensure units align with documented requirements.
🎫 Ticketing Providers
Jira, Linear, GitHub Issues, GitLab Issues
Epic and ticket management. The /elaborate-ticket-sync skill creates epics, tickets, and links that mirror the AI-DLC intent/unit structure in your ticketing system.
💬 Comms Providers
Slack, Teams, Discord
Notifications and updates. Progress updates, blocker alerts, and completion notifications are sent to configured channels.
Provider schemas are defined in plugin/schemas/providers/ and include additional stack-level schemas for operations: alerting, compute, infrastructure, monitoring, operations, packaging, pipeline, and secrets.
